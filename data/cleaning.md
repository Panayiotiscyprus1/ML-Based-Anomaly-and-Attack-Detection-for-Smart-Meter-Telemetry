# Cleaning & Preprocessing — Process and Findings

**Project:** ML-Based Anomaly and Attack Detection for Smart-Meter Telemetry
**Stage:** 2 (Cleaning & preprocessing)
**Module:** `src/cleaning.py`

This document explains *what* the cleaning does, *why* each decision was made, and *what
we found in the data*.

---

## 1. Philosophy

Three principles govern data handling in this project:

- **Raw data is immutable.** Files in `data/raw/` are exactly as received from the
  platform and are never edited in place. Every transformation reads *from* raw and
  writes its output *elsewhere*.
- **Processed data is reproducible, not source.** `data/processed/` holds cleaned
  outputs that can always be regenerated by re-running the pipeline.
- **Both raw and processed data are git-ignored.** The data is sensitive operational
  data from a water utility (EOA). The repository contains **code and summary reports
  only** — never raw or processed readings.

A fourth, structural principle: only the **loading** stage knows
where data comes from. As of 24/06/26 is CSV files; later it will be a read-only database
query. When the source changes, only `data_loading.py` changes — cleaning and everything
downstream are unaffected.

## 3. Input contract

`clean_meter()` consumes the output of `data_loading.load_meter_csv()`:

- a `DatetimeIndex` named `dataTime`, sorted ascending (the raw export is newest-first
  and is sorted on load);
- columns: `deviceId`, `flow` (cumulative forward-flow counter, m³), `signalCsq`,
  `isWaring`.

`flow` is a **cumulative** counter. Hourly consumption is derived by differencing.

## 4. The cleaning pipeline (stage by stage)

Each step below lists *what* it does and *why*.

### 4.1 Commissioning trim
**What:** drop the leading run of readings before the meter's first real flow
(`first_flow_time` = first hour the cumulative counter increases).
**Why:** It seems that newly installed meters are powered on and report *before* being plumbed into
the pipe. During that period the counter sits flat at zero — this is "the meter was not
yet measuring," not "the building used no water." Leaving it in would contaminate the
meter's notion of normal (it would look idle for a period when it was not even connected).

### 4.2 Consumption differencing
**What:** `consumption = flow.diff()`.
**Why:** the counter is cumulative; the quantity of interest is per-hour usage, which is
the difference between consecutive readings.

### 4.3 Counter-rollback handling (two-layer)
**What:** where `consumption < 0` (the counter decreased), set that hour's consumption to
`NaN` **and** record a `counter_rollback` event (timestamp, meter, magnitude, evidence).
**Why:** a cumulative counter should never decrease; a decrease means a reset,
replacement, or data error. The negative diff is not "negative water usage" — it is a
corrupt value. Feeding it to the detector would wreck rolling statistics and feature
scaling. But the rollback *is* a real anomaly, so it is not discarded — it is caught by a
deterministic rule and emitted as an event. **Principle: deterministic, certain anomalies
are caught by rules here; the ML methods are reserved for subtle, ambiguous ones.**
We replace with `NaN` (not 0) because the true usage during a reset is *unknown*, and
`NaN` says exactly that.

### 4.4 Hourly reindex
**What:** reindex onto a continuous 1-hour grid; missing hours become `NaN`.
**Why:** two reasons. (1) It makes gaps **explicit and locatable** — before reindexing a
missing hour is invisible (the row simply does not exist); after, it is a `NaN` row we can
count and later emit as a gap event. (2) It keeps the time axis **regular**, so a "rolling
24-hour window" always means 24 hours — windowed features and rolling statistics would
silently break on an irregular index.

### 4.5 Coverage-based service-start trim (scalable)
**What:** compute reporting density (fraction of non-`NaN` hours over a centred 7-day
window); find the first time density reaches a threshold (`service_start`); trim
everything before it.
**Why:** some meters report only sporadically for a long initial period before entering
consistent service (see meter …899 below). This is a *different* problem from
commissioning: commissioning is "reporting but flat-zero"; this is "barely reporting at
all." A hardcoded per-meter start date does not scale to a 2,031-meter fleet, so the trim
is **rule-based on density** — the same rule for every meter. It is a **no-op on healthy
meters** (dense from the start → nothing trimmed) and only acts on degenerate ones.

### 4.6 Usability classification
**What:** from the post-trim reporting rate, classify each meter `good` / `sparse` /
`unusable`.
**Why:** some meters are still mostly missing even after trimming — no amount of cleaning
rescues them. The flag lets the pipeline **auto-exclude** hopeless meters at fleet scale,
instead of someone discovering meter-by-meter that results are noise.

### 4.7 Gap-boundary diff guard
**What:** a consumption value computed across a transmission gap is set to `NaN` (the diff
spanned missing time and cannot be attributed to a single hour).
**Why:** see finding 5.2 below — this fixes a class of false spikes.

## 5. Data-quality findings

These are real characteristics of the data, discovered during Week 2.

### 5.1 Per-meter coverage and behaviour varies enormously
The five development meters differ sharply in how much they report and how much they use:
 
- Raw histories span roughly **8,000–10,400 hourly readings** (~11–14 months).
- **Zero-consumption (idle) proportion ranges from ~25% to ~96%** across meters — some
  sites are near-idle with occasional bursts, others are active most hours. This is real
  behavioural signal (it foreshadows the residential / business / industry distinction)
  and it is the reason the baseline detector must handle **zero-inflation** explicitly.
- Counter rollbacks are rare (**0–5 per meter**).
- Transmission gaps are mostly few per meter. Meter …899 had a long sparse *preamble*
  before service start, now trimmed (see 5.3); its cleaned series is not gap-heavy.


### 5.2 Gap-boundary diff artifact (fixed)
**Observed:** when a meter fails to transmit for a period, its cumulative counter keeps
advancing internally; on reconnection, the first reading reflects *all* the water used
during the silence. Differencing then attributes that entire multi-hour chunk to a single
hour, producing a large false spike (clearly visible as a lone ~0.54 m³ peak on meter
…909 around April 2026, far above its normal ~0.01–0.1 m³).
**Why it matters:** it is not a real one-hour anomaly, but it is the most extreme value in
the series — it would be a dramatic false positive and would distort the scale of all
nearby readings (contaminating rolling statistics and feature scaling).
**Fix:** consumption is set to `NaN` when the preceding hour was a gap, because the hourly
breakdown of usage across a gap is unknowable. The fact that usage occurred is not denied;
we simply refuse to misattribute it to one hour.
 
### 5.3 Meter …899 — long sparse preamble (trimmed), then a low-usage profile
**Observed (raw):** meter …899 reported almost nothing for its first ~9 months — of the
6,541 hours before ~July 2025, only 8 were real readings (>99% missing). Its `first_flow`
anchored on a tiny early blip in late 2024, months before real service began.
**Handling:** the coverage-based service-start trim (4.5) detected this automatically and
trimmed to where reporting becomes consistent (~July 2025).
**After trimming:** the meter runs cleanly from ~July 2025 onward (~11 months) with no
large missing blocks (usability flag: `good` — high *reporting* rate). It is a **very
low-usage meter (~94% idle hours)** with sparse, distinct consumption bursts — i.e. an
extreme of the zero-inflation challenge and a likely "residential / low-use" candidate for
the behavioural clustering, not a coverage problem. Note the distinction: a high *zero*
rate (idle) is not a high *missing* rate (unreported) — this meter is mostly idle, not
mostly missing.

## 6. Parameters (tunable policy)

`clean_meter()` exposes its policy as parameters with conservative defaults:

| Parameter | Default | Meaning |
|---|---|---|
| `reindex_hourly` | `True` | put the series on a continuous hourly grid |
| `rollback_to` | `"nan"` | replacement for rollback hours (`nan` is honest; `zero` optional) |
| `coverage_window_hours` | `168` | rolling window (7 days) for reporting-density |
| `density_threshold` | `0.5` | reporting fraction that counts as "in service" |
| `usable_min_reporting` | `0.70` | post-trim reporting rate below which a meter is `unusable` |

These defaults were chosen to **under-trim rather than over-trim** (when unsure, keep data
and flag it). They should be re-tuned against the full fleet once database access is
available.

---

## 7. Output

`clean_meter()` returns a `CleanResult` with:

- `.df` — the cleaned, hourly-indexed series (adds a `consumption` column);
- `.report` — a per-meter data-quality dict (rows trimmed, rollbacks, gaps, zero-rate,
  reporting rate, usability, consumption stats);
- `.rollback_events` — rule-detected rollback anomalies, ready for the event-output layer.

`report_table([...])` assembles the per-meter reports into one table — the data-quality
report deliverable.

Cleaned series are saved to `data/processed/` (git-ignored). The aggregate report is saved
to `reports/` (committed).
